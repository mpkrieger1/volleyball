## Live Play engine

### Back-row attacks (deferred from Sprint 31 Task 31.3)

**Why deferred:** the rally FSM in `workers/src/sim/rally.ts` does not model
back-row attacks. Today, any back-row attacker selection produces a
`rotation_violation` point loss (Sprint 4 guard rail). The Sprint 31 spec
included a "back-row attack frequency cap" feature with per-hint values
(aggressive=0.45, balanced=0.30, defensive=0.20) — these knobs exist in
`shared/src/sim/live/positionalRules.ts` HINT_TABLE but no code reads them.

**To ship in v1.2:**
1. Add a probabilistic branch in the rally FSM's `pickAttacker` that — when
   `useLivePositionalRules` is true — chooses between front-row and back-row
   attackers based on a target ratio derived from per-set running stats.
2. Add per-set back-row attack counter to LiveMatchState; reset at set start.
3. Probability table for back-row attack outcomes (kills, errors, blocks)
   should be DISTINCT from front-row (back-row attacks have lower kill rate).
4. Wire HINT_TABLE.backRowAttackCap (already locked: 0.20/0.30/0.45) into
   the cap check.
5. Update CLAUDE.md "From Sprint 31" — remove the back-row deferral note.

**Rough effort:** ~4 hours engine work + Monte Carlo calibration tests.

**Real-world impact:** D-I women's volleyball runs ~25-35% of attacks from
back-row at the highest level (pipe attacks especially). Without this, live
mode under-represents back-row offense.

### Per-rotation tactical hints (deferred from Sprint 31 Task 31.1)

**Why deferred:** original Q9=C spec proposed 18 toggles (6 rotations × 3
hint types per system). Cut to one-per-system per user revision during
Sprint 31 design.

**To ship in v1.2:** if/when granular control becomes valuable, extend
`TacticalHint` to a 6-element array indexed by current rotation. The
HINT_TABLE lookup in `positionalRules.ts` becomes rotation-aware.

**Estimated effort:** ~2 hours.

---

### "Show opponent rotation" toggle (deferred from Sprint 31 Task 31.2)

**Why deferred:** RotationTracker only renders the user team's rotation.
Spec mentioned a "show opponent rotation" toggle (default off). Not built.

**To ship in v1.2:** add a setting in `useSettingsStore` for
`showOpponentRotation: boolean`. LivePlayHub renders a second
`<RotationTracker>` for the opponent when toggled on. Already-built
component supports it via the `team` prop.

**Estimated effort:** ~30 min.

## Live Play UX (deferred from Sprint 30/31)

### Coach AI suggestions

Expected v1.2 feature: passive coaching suggestions — "Opponent is on a 4-0
run. Consider calling timeout?" Surfaced as a non-blocking toast in the
Coaching Strategy pane. Triggers reuse the smart-pause registry.

### "Watch your live match back" replay

`Match.coachActionsJson` has been populated since Sprint 29; `Match.pbpJson`
is the rally event log. A v1.2 replay viewer can interleave both into a
play-by-play timeline with coach-action banners. No new data needed.

### Coach portrait + speech bubble during timeouts

Cosmetic. Real coaches should appear during the skill-talk modal. Requires
coach-portrait assets + a small visual layer in `SkillTalkModal.tsx`.

### Per-player fatigue surfaced in UI

Sprint 30 added an internal fatigue model used for AI sub decisions. Not
yet visible to the user. v1.2 could expose per-player fatigue in the sub
picker (color-coded badge).

### Timeout types

NCAA actually distinguishes 60s timeouts and 75s media timeouts. Cosmetic
distinction; v1.2 polish.

## Test infrastructure

### `tests/integration/save/save10Seasons.test.ts`

Sprint 23 deliverable that never landed. Sprint 31 Task 31.5 substituted a
focused per-match size assertion in lieu of a 10-season aggregate. v1.2
can revisit if save-file growth becomes a concern.

### Live mode in calibration

By design, live mode is excluded from `npm run test:calibration:full` (the
gating discriminator pattern from Sprint 31 keeps it sim-only). If a v1.2
need arises to calibrate live behavior against benchmarks, a parallel
`test:calibration:live` suite would replicate the pattern with synthetic
coach inputs.
